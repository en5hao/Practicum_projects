# Practicum_projects
Разработка приложения Stets Home v1.0
Цель проекта: Спроектировать и прототипировать приложение для управления умным домом Stets с акцентом на энергосбережение, совместный доступ и интуитивное управление.

Основные цели исследования и проектирования:

Определить минимальный жизнеспособный набор функций (MVP) для запуска.

Спроектировать удобный интерфейс для широкого круга пользователей.

Создать четкую архитектуру данных и взаимодействия для будущей разработки.

Обеспечить возможность совместного управления устройствами членами семьи.

Реализовать ключевую бизнес-функцию — управление энергосберегающим режимом устройств.

Библиотеки/Инструменты, использованные на этапе проектирования:

Диаграммы: Miro, Draw.io, PlantUML (для DFD, ER-диаграмм, карт историй)

Прототипирование: Figma (с использованием готовой дизайн-системы Stets)

Документирование: Confluence, Google Docs, Markdown (для ПМИ, словаря данных)

Моделирование API: Swagger/OpenAPI, Postman (для профиля API)

Презентация: OBS Studio, Zoom (для записи видео-презентации)

Этапы работы и содержание файлов проекта
Этап 1: Анализ требований и проектирование функциональности

Файлы: 3_Карта_пользовательских_историй, 7_Figma_прототипы (черновики)

Что делалось: На основе вводных данных были выделены пользовательские роли (Владелец, Член семьи, Гость), определены эпики и сформулированы пользовательские истории. Истории были приоритизированы для MVP (например, управление энергосбережением) и бэклога (например, авторизация через социальные сети). Проведена проверка историй по критерию INVEST.

Этап 2: Проектирование данных и взаимодействий

Файлы: 5_DFD_ER-диаграмма, 6_Словарь_данных_приложения, 2_Моделирование_профиля_API

Что делалось:

ER-диаграмма и Словарь данных: Определены ключевые сущности (Пользователь, Дом, Комната, Устройство, Сценарий) и их атрибуты. Установлены связи (например, в одном Доме может быть от 0 до 10 комнат, устройство принадлежит пользователю). Это "модель данных" приложения.

DFD (Data Flow Diagram): Визуализированы потоки данных между системами (клиентское приложение, сервер, база данных, внешние сервисы уведомлений), чтобы понять, как информация передается при регистрации, добавлении устройства и т.д.

Профиль API: Спроектированы ключевые конечные точки REST API (например, POST /api/devices для добавления устройства, PUT /api/devices/{id}/power-saving для управления режимом), определены форматы запросов и ответов.

Этап 3: Проектирование интерфейса и навигации

Файлы: 1_Диаграмма_структуры_интерфейса, 7_Figma_прототипы

Что делалось:

Структура интерфейса: Создана иерархическая карта экранов приложения (от Splash-экрана до экрана настроек устройства), которая легла в основу навигации в прототипе.

Интерактивный прототип в Figma: На основе готовой дизайн-системы Stets собраны все ключевые экраны MVP, связанные между собой кликабельными переходами. Это включает экраны онбординга, главный экран с комнатами, процесс добавления устройства (QR/ручной ввод), экран управления устройством, раздел автоматизации и профиль.

Этап 4: Планирование тестирования и финализация

Файлы: 4_ПМИ (Программа и методика испытаний)

Что делалось: Разработан подробный документ для QA-специалистов. В ПМИ описаны:

Цели испытаний: Проверка работоспособности MVP.

Объекты испытаний: Прототип интерфейса, спроектированная логика API.

Методика: Пошаговые сценарии тестирования для каждой пользовательской истории из MVP (например, "Сценарий Т-1.1: Успешная регистрация").

Критерии приемки: Четкие условия, при которых функция считается работающей (сформулированы в стиле "Given-When-Then").

Требования к окружению: Figma для прототипа, Postman для проверки API.

Этап 5: Презентация и передача результатов

Итоговое действие: Записано видео-презентация (20-25 минут).

Что делалось: Системный аналитик, опираясь на шаги из ПМИ, провел "виртуальный тур" по прототипу: продемонстрировал главные пользовательские сценарии (регистрация, добавление устройства, включение энергосберегающего режима), объяснил логику, заложенную в диаграммах (DFD, ER), и показал, как документы (карта историй, ПМИ) связаны с финальным прототипом. Это был итоговый отчет о выполненной работе по проектированию.

Вывод: Проект представляет собой комплексную работу системного аналитика — от анализа требований и проектирования данных до создания интерактивного прототипа и полного пакета документации, готового к передаче командам разработки и тестирования.
